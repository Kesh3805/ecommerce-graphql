Variants require understanding JSON array encoding. The new template can be handed to a non-technical user and filled in like a normal spreadsheet.
2. Onboarding mutation flow — stepwise vs monolithic
How a new product gets from zero to published.

Legacy flow

Monolithic form submit

A single large REST payload bundles product identity, tax flags, fulfillment settings, media, category assignment, WYSIWYG description, disclaimer text, cancelability flags, and variant data all in one submit.


Product listing and filtering is handled by dynamically assembled SQL strings with direct interpolation of country, brand, category, and attribute filters — building the query text at runtime.


Variants use FIND_IN_SET() on comma-packed attribute IDs. No atomic transaction wraps the create path.

New flow

Typed, stepwise mutations
1
createProductWrites Product, upserts Brand, persists Metafields + CountryAvailability — all in one transaction
2
attachProductMediaUpdates media_urls and primary_image_url on the product row
3
addProductOptionWrites ProductOption + OptionValue rows with position uniqueness enforced
4
createVariantWrites Variant + InventoryItem — SKU uniqueness enforced at DB level
5
setInventoryLevelWrites InventoryLevel + immutable InventoryAdjustment journal entry
6
publishProductSets status + published_at, triggers ES sync
Every write step has a clear transaction boundary. If variant creation fails, the product record stays intact and valid. Each step has a defined table boundary, making debugging, rollback, and QA testing precise. The legacy monolith either succeeds completely or leaves partial state with no clean rollback path.
3. Read performance — ES read model vs runtime joins
How storefront queries are served under load.

Legacy pattern

  • Product listing builds large SQL strings at runtime
  • Filters for country, brand, category, attributes joined per query
  • Variant lookups use FIND_IN_SET() — not index-friendly
  • Every storefront request hits the relational DB directly
  • No caching layer — no request deduplication

New pattern

  • Elasticsearch is the primary read store for storefront queries
  • PostgreSQL is the write source of truth and automatic read fallback path
  • TTL in-memory cache on products, stores, collections paths
  • In-flight request deduplication — concurrent identical requests share one call
  • DB fallback kicks in automatically if ES is unavailable

4. Database design — integrity by construction
What the schema enforces at the database layer, not just in application code.

Legacy DB

  • products table used MyISAM — no transactions, no FK support
  • Variant options stored as comma-packed text IDs in one column
  • Country pricing spread across 3+ join tables per query
  • No unique constraints on SKU, handle, or option position
  • No inventory adjustment journal — stock changes leave no audit trail
  • Application code solely responsible for consistency

New DB model — 28 typed entities

  • PostgreSQL (MVCC + WAL) — full ACID transactions on all write paths
  • Relational option model: ProductOption + OptionValue rows
  • ProductCountryAvailability — one indexed table, no join chain
  • Unique: product handle, variant SKU, option position, inventory level per location
  • InventoryAdjustment — immutable journal for every stock change
  • Row-level pessimistic locks for concurrent inventory operations

5. Summary — three concrete wins
What this migration delivers, practically.
Onboarding is self-service. The new import template uses slugs, names, pipe-separated values, and flat rows. Any ops or merchant team member can fill it in. The legacy template requires DB ID lookups and JSON blob construction before a single product can be submitted.
Reads are an order of magnitude faster. Storefront pages go from 800–1,500ms (legacy live join queries) to 200–300ms on first load and 20–80ms on cache hit. This is architectural — not a query tweak.
Data can't go wrong silently. Unique constraints, FK-backed entity relations, transactional writes, and row-level inventory locking mean the database enforces correctness. The legacy MyISAM setup left consistency entirely to application code — any bug, crash, or partial write left undetectable corrupt state.
What is not being compared: Fields like tax configuration, disclaimer text, and video metadata exist in the legacy template but are not yet in the new one — these are acknowledged incremental additions, not regressions. The comparison above is limited to what both systems do today.
